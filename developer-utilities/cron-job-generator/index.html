Demystifying the 5-Field Linux Cron Architecture

Every standard Linux crontab expression is a space-delimited string of exactly five positional fields. Knowing what each field controls — and which wildcard operators it accepts — is the foundation of confident cron literacy. The table below maps each position to its numeric bounds, supported operators, and a concrete usage example.

# Field Allowed Range Operators Example & Meaning
1 Minute 0 – 59 */-, */15 — every 15 minutes
2 Hour 0 – 23 */-, 0,12 — at midnight and noon
3 Day of Month 1 – 31 */-,? 1,15 — 1st and 15th of each month
4 Month 1 – 12 */-, 3-6 — March through June
5 Day of Week 0 – 7 (0 & 7 = Sun) */-,? 1-5 — Monday through Friday

Operator quick-reference: The asterisk (*) matches every valid value in its field. The slash (/) defines a step interval — */5 in the Minute field fires every 5 minutes. The hyphen (-) sets an inclusive range between two values. The comma (,) enumerates discrete values within a single field, letting you target multiple non-sequential points without duplicating crontab lines.

Frequently Asked Questions

Practical answers to the edge cases developers encounter most when working with cron expressions.

What is the operational difference between 5-field Linux cron and 6-field Quartz syntax? +

Standard POSIX/Vixie cron uses five fields — Minute, Hour, Day of Month, Month, Day of Week — and is the format natively supported by /etc/crontab, crontab -e, and virtually every Linux-based scheduler, including AWS EventBridge, GCP Cloud Scheduler, and GitHub Actions scheduled workflows.

Quartz Scheduler — ubiquitous in Java enterprise environments and tools like Spring Framework, Jenkins, and Quartz.NET — prepends a mandatory Seconds field, producing a 6-field (or optionally 7-field, with a trailing Year column) format. Its field order reads: Seconds Minutes Hours Day-of-Month Month Day-of-Week [Year].

This generator targets the universal 5-field Linux standard. If you are configuring a Quartz-based system, simply prepend your desired seconds value (e.g., 0) to the left of any expression this tool produces to achieve full compatibility.

How do non-standard macro shorthands like @daily, @hourly, and @reboot work? +

Many modern cron implementations — including Vixie cron, cronie, and systemd timers in compatibility mode — support a set of @ nickname macros that act as human-readable aliases for common 5-field expressions:

  • @reboot — runs once at system startup; has no direct 5-field equivalent
  • @yearly / @annually — equivalent to 0 0 1 1 *
  • @monthly — equivalent to 0 0 1 * *
  • @weekly — equivalent to 0 0 * * 0
  • @daily / @midnight — equivalent to 0 0 * * *
  • @hourly — equivalent to 0 * * * *

These macros are not part of the POSIX standard and are unsupported in many cloud schedulers, containerised environments running minimal cron daemons, and all Quartz-based systems. For maximum portability, always prefer the explicit 5-field expressions this tool generates over shorthand macros, and verify @reboot support independently against your target daemon.

Are there system memory constraints when processing continuous automated cron tasks? +

At the OS scheduler level, cron itself is remarkably lightweight — the daemon typically holds under 1 MB of resident memory and forks a fresh child process per job rather than maintaining long-lived workers. The memory constraint to plan for is the spawned job, not the scheduling daemon.

  • Concurrency collisions — a job scheduled every 5 minutes that takes 8 minutes to run will accumulate overlapping instances. Use flock lock files or tools like run-one to prevent stacking and runaway memory consumption.
  • ulimit ceilings — per-user memory limits defined in /etc/security/limits.conf apply to all cron-spawned processes. Large ETL or inference jobs may require explicit ulimit -v overrides inside the crontab command string.
  • Container environments — Kubernetes CronJobs inherit the pod's resource requests and limits. Always define explicit memory requests to prevent OOMKill termination of scheduled tasks during cluster contention events.
